Домашнее задание по теме 4
Дисциплина: Управление производительностью приложений.

Тема: Причины деградации производительности.

Форма проверки: задание с самопроверкой, это задание преподаватель не проверяет, но мы можем выборочно посмотреть некоторые работ на курсе.

Примерное время выполнения: 3 часа.

Цель задания:

Применить знания о причинах деградации производительности к вашему собственному проекту.
Инструменты для выполнения задания:

Java (любая IDE, например, IntelliJ IDEA или Eclipse);
JDK 8 или выше;
VisualVM (для анализа GC-логов и профилирования);
JMH (для микробенчмарков).
Правила приёма работы:

Выполняйте задание по этапам.
Разместите ссылку на выполненное задание в личный кабинет. Убедитесь, что доступ к документу открыт.
Дедлайн: рекомендуем выполнить задание к следующему вебинару по дисциплине.

 

Описание задания:
Выберите 3 категории из списка ниже и для каждой:

Опишите, как эта проблема может проявиться в вашем проекте или проявляется.
Обоснуйте, почему вы опасаетесь / не опасаетесь этой проблемы.
Предложите, как её можно обнаружить (инструменты: JMeter, Micrometer, VisualVM, логи и т.д.) или предотвратить. Если проблема есть, проложить скрины с нахождением проблемы и описанием как решить.
Список категорий

Аппаратный уровень
– Нехватка CPU (например, при парсинге большого XML/JSON)
– Дисковый I/O (запись логов, кэш на диск)
– Сетевая пропускная способность (частые запросы к внешнему API курсов валют)

Контекстное переключение (context switching)
– Слишком много потоков (например, по одному на каждый курс валюты)
– Использование newCachedThreadPool() без ограничений

Архитектурный выбор
– Почему вы выбрали монолит или микросервисы?
– Есть ли в вашем проекте тесно связанные данные, которые “страдают” от разделения?
– Как обеспечивается согласованность (например, актуальность курса при конвертации)?

Алгоритмический уровень
– Какой алгоритм используется для обработки данных?
– Может ли он деградировать при росте объёма данных?
– Есть ли более эффективная альтернатива?

Среда выполнения
– Настройки JVM (GC, heap)
– Влияние контейнеризации (Docker memory/CPU limits)

 

Оформление отчёта:
Составьте отчет, в котором:

Укажите тип приложения (например, многопоточный парсер курсов валют, банковский сервис, торговый бот и т.д.).
Кратко опишите основную функциональность (что делает программа, какие данные обрабатывает).
Укажите ключевые технологии (Spring Boot, Java 21, PostgreSQL, REST API, многопоточность и т.п.).
Отметьте, является ли архитектура монолитом или микросервисной.
Упомяните, какие операции в приложении могут быть ресурсоёмкими (парсинг, сетевые вызовы, тяжёлые вычисления, работа с БД и др.).
Ссылку на отчет прикрепите в личном кабинете на платформе.

Для каждой категории вы чётко указали, где именно в проекте может быть проблема.
