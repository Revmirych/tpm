Домашнее задание по теме 8
Дисциплина: Управление производительностью приложений.

Тема: Производительность реляционных и нереляционных БД.

Форма проверки: задание с самопроверкой, это задание преподаватель не проверяет, но мы можем выборочно посмотреть некоторые работ на курсе.

Имя преподавателя: Василий Москалев

Примерное время выполнения: 3 часа.

Цель задания:

произвести обоснованный выбор типа базы данных на основе реальных требований проекта.
Инструменты для выполнения задания:

Java (любая IDE, например, IntelliJ IDEA или Eclipse);
JDK 17 или выше;
Spring Boot (для реализации адаптеров и инфраструктурных слоёв);
Spring Data JPA (для работы с реляционной СУБД, например PostgreSQL);
Spring Data MongoDB (для работы с нереляционной СУБД, например MongoDB);
JUnit / Mockito (для юнит-тестирования доменного ядра без Spring);
Браузер с DevTools (для проверки работы REST-адаптера);
PostgreSQL (или другая реляционная БД);
MongoDB (или другая документная БД);
Средства логирования.
Правила приёма работы:

Выполняйте задание по этапам.
Разместите ссылку на выполненное задание в личный кабинет. Убедитесь, что доступ к документу открыт.
Дедлайн: рекомендуем выполнить задание к следующему вебинару по дисциплине.

 

Описание задания:
Определите ключевую сущность вашего проекта (например, ExchangeRate, Review, UserProfile) и типичные сценарии работы с ней:
частота чтения/записи
потребность в связях с другими сущностями
необходимость транзакций или гибкости схемы
Реализуйте хранение этой сущности в двух СУБД:
Реляционной (например, PostgreSQL через Spring Data JPA)
Нереляционной (например, MongoDB через Spring Data MongoDB)
Проведите сравнительные замеры для следующих операций:
Вставка одной записи и пакетная вставка (например, 100–1000 записей)
Чтение одной записи по ключу
Чтение списка с фильтрацией (например, «все отзывы за неделю» или «курсы валют за день»)
Проанализируйте результаты:
Сравните latency и throughput
Оцените сложность реализации (например, JOIN vs вложенные документы)
Отметьте, где возникли узкие места
Сделайте вывод:
Какая СУБД лучше подходит для вашего сценария и почему?
Какие архитектурные компромиссы (согласованность, масштабируемость, гибкость) вы готовы принять?
 

Оформление отчёта:
Работа должна позволять оценить:

корректность выбора сущности и сценариев её использования (чтение, запись, фильтрация);
корректность реализации доступа к данным в реляционной и нереляционной СУБД;
корректность и сопоставимость замеров производительности;
различия в latency и throughput для разных типов операций;
сложность модели данных и запросов в каждой СУБД;
архитектурные компромиссы, выявленные в ходе сравнения.
Ссылку на отчёт необходимо прикрепить в личном кабинете на платформе.

 

Чек-лист самопроверки:
Выбрана и описана ключевая сущность проекта и сценарии работы с ней.
Реализовано хранение данных в реляционной СУБД.
Реализовано хранение данных в нереляционной СУБД.
Проведены замеры вставки одиночных и пакетных записей.
Проведены замеры чтения по ключу.
Проведены замеры чтения с фильтрацией.
Сравнены показатели latency и throughput.
Проанализирована сложность реализации запросов и модели данных.
Сделан вывод о применимости каждой СУБД для выбранного сценария.
Написан краткий отчет с описанием использованных инструментов и результатов.
