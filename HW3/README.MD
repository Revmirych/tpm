Домашнее задание по теме 3
Дисциплина: Управление производительностью приложений.

Тема: Инструменты профилирования, мониторинга и тестирования.

Форма проверки: задание с самопроверкой, это задание преподаватель не проверяет, но мы можем выборочно посмотреть некоторые работы на курсе.

Примерное время выполнения: 3 часа.

Цель задания:

Научиться интегрировать инструменты профилирования, мониторинга и тестирования в Java-приложения;
Научиться проводить нагрузочное тестирование собственного веб-сервиса и анализировать его поведение под нагрузкой с использованием Apache JMeter.
Инструменты для выполнения задания:

Java (любая IDE, например, IntelliJ IDEA или Eclipse);
JDK 8 или выше;
JMeter (для нагрузочного тестирования).
Правила приёма работы:

Выполняйте задание по этапам.
Разместите ссылку на выполненное задание в личный кабинет. Убедитесь, что доступ к документу открыт.
Дедлайн: рекомендуем выполнить задание к следующему вебинару по дисциплине.

 

Описание задания:
Необходимо разработать в JMeter набор сценариев, отражающих типичные пользовательские взаимодействия с API: как операции чтения (например, получение списка сущностей), так и операции записи (создание, обновление — при наличии).

При этом важно учитывать реалистичность запросов: использовать разные параметры, валидировать ответы и моделировать поведение нескольких одновременных пользователей.
 

Оформление отчёта:
Тест должен позволять оценить:

устойчивость сервиса при нагрузке,
время отклика ключевых эндпоинтов,
наличие ошибок или деградации производительности по мере увеличения числа запросов.
В результате необходимо предоставить тестовый сценарий в формате JMeter (.jmx) и краткий аналитический вывод: как ведёт себя система под нагрузкой, какие метрики были зафиксированы, и есть ли предпосылки для оптимизации (например, медленные запросы, ошибки, резкий рост задержек).

Ссылку на отчет прикрепите в личном кабинете на платформе.

 

Чек-лист самопроверки:
В тесте присутствуют как минимум два разных типа запросов (например, GET + POST или два разных GET-сценария).
Есть валидация ответов: проверка статуса (200 OK) и/или содержимого (например, наличие ожидаемого поля в JSON).
Настроено моделирование нескольких пользователей (Thread Group с >1 потоком).
Учтён плавный запуск нагрузки — не все пользователи стартуют одновременно.
Используются разные входные данные при необходимости (например, через CSV или переменные), чтобы избежать кэширования или неестественного поведения.
Написан краткий отчет с описанием использованных инструментов и результатов тестирования.
